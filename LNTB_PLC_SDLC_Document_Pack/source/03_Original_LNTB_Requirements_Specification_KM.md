# LNTB — ប្រព័ន្ធកសិកម្មឆ្លាតវៃសម្រាប់ដំណាំប៉េងប៉ោះឈើរី
## ឯកសារតម្រូវការគម្រោង (Requirements Specification Document)

**ក្រុម៖** LNTB (Leading Next-Gen Tomato Business)
**បច្ចេកវិទ្យា៖** Laravel (Backend) · Flutter (Mobile App) · Arduino/C++ លើ ESP32 (Firmware)

---

## ១. សេចក្តីផ្តើម

### ១.១ គោលបំណងឯកសារ
ឯកសារនេះកំណត់ពីអ្វីដែលផ្នែកនីមួយៗនៃប្រព័ន្ធ LNTB ត្រូវធ្វើ និងលក្ខខណ្ឌដែលវាត្រូវដំណើរការក្រោម។
វារួមបញ្ចូលទាំង តម្រូវការមុខងារ (Functional Requirements), តម្រូវការក្រៅមុខងារ (Non-Functional
Requirements), តម្រូវការផ្នែករឹង, តម្រូវការផ្នែកទន់ និងតម្រូវការទិន្នន័យ សម្រាប់ទាំង ៣ ផ្នែក៖
Firmware (IoT), Backend (Laravel) និង Mobile App (Flutter)។

### ១.២ វិសាលភាព
ប្រព័ន្ធនេះជួយកសិករដាំប៉េងប៉ោះឈើរីក្នុងផ្ទះសំណាញ់ (Net house) នៅកម្ពុជា ឱ្យអាចធ្វើស្វ័យប្រវត្តិកម្មប្រព័ន្ធស្រោចស្រព
តាមដានចំណាយទឹក/ភ្លើងជាក់ស្តែង និងកំណត់ភាពទុំរបស់ផ្លែដោយប្រើសេនស័រ និង AI Camera
ដែលគ្រប់គ្រងទាំងអស់តាមរយៈកម្មវិធីទូរស័ព្ទដៃជាភាសាខ្មែរ។

### ១.៣ និយមន័យពាក្យ
| ពាក្យ | អត្ថន័យ |
|---|---|
| Net house | ផ្ទះសំណាញ់សម្រាប់ដាំដុះប៉េងប៉ោះឈើរី |
| Sensor node | ឧបករណ៍ ESP32 + សេនស័រវាស់សំណើម/សីតុណ្ហភាព/ពន្លឺ + Relay/ម៉ូទ័រទឹក |
| Camera node | ESP32-CAM សម្រាប់ពិនិត្យភាពទុំរបស់ផ្លែ |
| កសិករ (Farmer) | អ្នកប្រើប្រាស់ចម្បងនៃកម្មវិធីទូរស័ព្ទដៃ |

## ២. តម្រូវការទូទៅនៃប្រព័ន្ធ

- ប្រព័ន្ធនេះនឹងមានផ្នែកសំខាន់ៗចំនួន ៣ ធ្វើការរួមគ្នា៖ Firmware (Arduino/ESP32), Backend (Laravel)
  និង Mobile App (Flutter)។
- ប្រព័ន្ធនេះត្រូវអាចដំណើរការទោះបីជាមានសញ្ញា WiFi ខ្សោយ ឬមិនទៀងទាត់ តាមស្ថានភាពជាក់ស្តែងនៅតំបន់ជនបទក៏ដោយ។
- ប្រព័ន្ធនេះត្រូវប្រតិបត្តិក្នុងថវិកាផ្នែករឹងប្រហែល $១០៣.៥០ ក្នុងមួយ Net house (យោងតាមបញ្ជីធនធានដើម)។
- ប្រព័ន្ធនេះត្រូវគាំទ្រយ៉ាងតិច ១ Net house ក្នុងមួយគណនីកសិករ សម្រាប់ដំណាក់កាល MVP។

## ៣. តម្រូវការមុខងារ (Functional Requirements)

### ៣.១ Firmware (Arduino / ESP32) — FR-HW

| លេខកូដ | តម្រូវការ |
|---|---|
| FR-HW-01 | Sensor node ត្រូវអានតម្លៃសំណើមដីតាមចន្លោះពេលកំណត់ (លំនាំដើម៖ រៀងរាល់ ៦០ វិនាទី)។ |
| FR-HW-02 | Sensor node ត្រូវអានសីតុណ្ហភាព និងសំណើមខ្យល់ តាមរយៈ DHT22។ |
| FR-HW-03 | Sensor node ត្រូវអានកម្រិតពន្លឺ តាមរយៈ LDR។ |
| FR-HW-04 | Sensor node ត្រូវផ្ញើទិន្នន័យទៅ Laravel API ជាទម្រង់ JSON តាមរយៈ HTTPS។ |
| FR-HW-05 | Sensor node ត្រូវ Poll (សួរ) endpoint ពាក្យបញ្ជានៅ Laravel តាមចន្លោះពេលកំណត់ (លំនាំដើម៖ រៀងរាល់ ១០ វិនាទី) ដើម្បីពិនិត្យពាក្យបញ្ជាដែលមិនទាន់ដំណើរការ។ |
| FR-HW-06 | Sensor node ត្រូវបើក Relay (ម៉ូទ័រទឹក) នៅពេលទទួលពាក្យបញ្ជា `pump_on` និងបិទនៅពេលទទួល `pump_off`។ |
| FR-HW-07 | Sensor node ត្រូវបញ្ជា Servo (សំណាញ់ដំបូល) នៅពេលទទួលពាក្យបញ្ជា `servo_open` / `servo_close`។ |
| FR-HW-08 | Sensor node ត្រូវផ្ញើការបញ្ជាក់ (Acknowledge) ត្រឡប់ទៅ Backend វិញ នៅពេលពាក្យបញ្ជាត្រូវបានប្រតិបត្តិរួច។ |
| FR-HW-09 | Sensor node ត្រូវប្រើតម្លៃដែនកំណត់សំណើមដីដែលបានកំណត់ទុកជាមុនក្នុងតួឧបករណ៍ (Local Fallback) ដើម្បីស្រោចទឹកដោយស្វ័យប្រវត្តិ ក្នុងករណីមិនអាចទាក់ទង Backend បានលើសពេលកំណត់។ |
| FR-HW-10 | Camera node ត្រូវថតរូបផ្លែប៉េងប៉ោះតាមចន្លោះពេលកំណត់ ឬតាមការបញ្ជា។ |
| FR-HW-11 | Camera node ត្រូវផ្ញើរូបភាពដែលបានថត (ឬលទ្ធផលនៃការវិភាគភាពទុំ អាស្រ័យលើការសម្រេចចិត្តចុងក្រោយថាតើដំណើរការ AI នៅឯណា) ទៅកាន់ Laravel API។ |
| FR-HW-12 | Firmware ត្រូវរក្សាទុក WiFi credentials និង API endpoint នៅក្នុងឯកសារ config ដាច់ដោយឡែក មិនសរសេរជាប់នឹងកូដចម្បង។ |
| FR-HW-13 | Firmware ត្រូវភ្ជាប់ WiFi ឡើងវិញដោយស្វ័យប្រវត្តិ ក្រោយពេលការតភ្ជាប់ដាច់ ដោយមិនចាំបាច់ចុច Reset ដោយដៃ។ |

### ៣.២ Backend (Laravel) — FR-BE

| លេខកូដ | តម្រូវការ |
|---|---|
| FR-BE-01 | Backend ត្រូវផ្តល់ endpoint ដែលមានការផ្ទៀងផ្ទាត់ សម្រាប់ ESP32 ផ្ញើទិន្នន័យសេនស័រចូល។ |
| FR-BE-02 | Backend ត្រូវផ្តល់ endpoint សម្រាប់ ESP32 Poll ពាក្យបញ្ជាដែលមិនទាន់ដំណើរការ។ |
| FR-BE-03 | Backend ត្រូវផ្តល់ endpoint សម្រាប់ ESP32 បញ្ជាក់ថាបានប្រតិបត្តិពាក្យបញ្ជារួច។ |
| FR-BE-04 | Backend ត្រូវផ្តល់ endpoint សម្រាប់ទទួលលទ្ធផល/រូបភាពនៃការពិនិត្យភាពទុំ។ |
| FR-BE-05 | Backend ត្រូវគណនាបរិមាណទឹក (ម៉ែត្រគូប) និងអគ្គិសនី (គីឡូវ៉ាត់ម៉ោង) ដែលបានប្រើប្រាស់ជាក់ស្តែង ក្នុងមួយព្រឹត្តិការណ៍ស្រោចទឹក។ |
| FR-BE-06 | Backend ត្រូវដំណើរការ Job ស្វ័យប្រវត្តិប្រចាំថ្ងៃ ដើម្បីសរុបការប្រើប្រាស់ចូលទៅក្នុងតារាង `usage_records` ព្រមទាំងគណនាថ្លៃដើមជាដុល្លារ។ |
| FR-BE-07 | Backend ត្រូវផ្សព្វផ្សាយទិន្នន័យសេនស័រថ្មីៗទៅកាន់កម្មវិធី Flutter ជាបន្តបន្ទាប់ (Real-time) តាមរយៈ WebSocket (Laravel Reverb) សម្រាប់ Farm នីមួយៗ។ |
| FR-BE-08 | Backend ត្រូវផ្ញើសារជូនដំណឹង (FCM Push Notification) នៅពេលមានផ្លែប៉េងប៉ោះទុំរួចរាល់ដើម្បីប្រមូលផល។ |
| FR-BE-09 | Backend ត្រូវផ្ញើសារជូនដំណឹង នៅពេលកម្រិតសំណើមដីឡើងហួស ឬធ្លាក់ចុះក្រោមកម្រិតកំណត់សំខាន់។ |
| FR-BE-10 | Backend ត្រូវផ្តល់ការផ្ទៀងផ្ទាត់ដោយ Token (Sanctum) សម្រាប់កម្មវិធី Flutter។ |
| FR-BE-11 | Backend ត្រូវផ្តល់ endpoint សម្រាប់ស្វែងរក/ត្រងទិន្នន័យប្រវត្តិកំណត់ហេតុ (Logbook) តាមខែ/ឆ្នាំ/ចន្លោះកាលបរិច្ឆេទ។ |
| FR-BE-12 | Backend ត្រូវអនុញ្ញាតឱ្យកសិករបញ្ជាបើក/បិទប្រព័ន្ធស្រោចទឹកដោយដៃ តាមរយៈ API ដោយត្រូវមានការផ្ទៀងផ្ទាត់ជាមុន។ |
| FR-BE-13 | Backend ត្រូវកត់ត្រាថាតើអ្វី/នរណាជាអ្នកបញ្ជាឱ្យស្រោចទឹកម្តងៗ (`auto` ឬ `manual`) និងកសិករណាដែលបញ្ជាដោយដៃ។ |
| FR-BE-14 | Backend ត្រូវអនុញ្ញាតកំណត់អត្រាតម្លៃ ($/ម៉ែត្រគូប, $/គីឡូវ៉ាត់ម៉ោង) ដាច់ដោយឡែកសម្រាប់ Farm នីមួយៗ ដើម្បីប្រើក្នុងការគណនាថ្លៃដើម។ |

### ៣.៣ Mobile App (Flutter) — FR-APP

| លេខកូដ | តម្រូវការ |
|---|---|
| FR-APP-01 | កម្មវិធីត្រូវបង្ហាញអេក្រង់ចូលប្រើ (Login) ហើយផ្ទៀងផ្ទាត់ជាមួយ Laravel Sanctum endpoint។ |
| FR-APP-02 | កម្មវិធីត្រូវបង្ហាញ Dashboard ជាក់ស្តែង (Real-time) បង្ហាញកម្រិតសំណើមដី សីតុណ្ហភាព សំណើមខ្យល់ និងកម្រិតពន្លឺ។ |
| FR-APP-03 | កម្មវិធីត្រូវបង្ហាញស្ថានភាពដោយប្រើពណ៌ (បៃតង = ធម្មតា, ក្រហម = ត្រូវប្រុងប្រយ័ត្ន)។ |
| FR-APP-04 | កម្មវិធីត្រូវអនុញ្ញាតឱ្យកសិករបញ្ជាបើក ឬបិទប្រព័ន្ធស្រោចទឹកដោយដៃបាន។ |
| FR-APP-05 | កម្មវិធីត្រូវបង្ហាញស្ថានភាពប្រព័ន្ធស្រោចទឹកស្វ័យប្រវត្តិបច្ចុប្បន្ន (កំពុងបើក/បិទ, ពេលចុងក្រោយដែលបានស្រោច)។ |

  ការរចនាប្រព័ន្ធផ្តោតលើភាពសាមញ្ញ ជាងលក្ខណៈស្មុគស្មាញខ្ពស់។

**សម្មតិកម្ម (Assumptions)**
- Net house នីមួយៗមានសញ្ញា WiFi យ៉ាងតិចម្តងម្កាល។
- កសិករមានទូរស័ព្ទដៃដែលអាចដំណើរការកម្មវិធី Flutter ជំនាន់ថ្មីបាន។
- ESP32-CAM មួយគ្រឿងក្នុងមួយ Net house គឺគ្រប់គ្រាន់សម្រាប់ការសាកល្បងពិនិត្យភាពទុំ (មិនចាំបាច់គ្របដណ្តប់ ១០០% នៃផ្លែទាំងអស់)។
